# 023 — Single source of truth for pin definitions

Status: done

## Problem

Pin semantics are defined in 4 places that must be manually kept in sync:
1. Rust component structs (`eerie-core/src/component.rs`)
2. TypeScript symbol registry (`src/renderer/src/symbols/index.ts`)
3. File-to-UI pin name mappings (`src/renderer/src/utils/netlistBuilder.ts` lines 8-23)
4. SPICE netlist output (`spice-netlist`)

If any one diverges, components render with wrong pins or simulations use wrong nodes.

## Proposed fix

Define pin metadata once in Rust (names, directions, types per component kind) and
generate the TypeScript pin mappings via codegen. The symbol registry and netlist
builder should both consume the generated pin data rather than maintaining their own
hardcoded maps.

## Acceptance criteria

- [ ] Pin names/directions defined once in Rust
- [ ] TypeScript pin data generated by `pnpm codegen`
- [ ] `FILE_PIN_TO_UI` map in netlistBuilder.ts removed or generated
- [ ] Symbol registry consumes generated pin data
